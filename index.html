RxJava的本质

RxJava在Github的主页介绍:

RxJava is a Java VM implementation of Reactive Extensions: a library for composing asynchronous and event-based programs by using observable sequences

It extends the observer pattern to support sequences of data/events and adds operators that allow you to compose sequences together declaratively while abstracting away concerns about things like low-level threading, synchronization, thread-safety and concurrent data structures.

RxJava是响应式扩展的Java VM实现：一个通过可观察序列来组成异步的、基于事件的程序的库

RxJava扩展了观察者模式，用以支持数据/事件的序列，并增加了很多操作符，让你把众多事件序列简洁地组合在一起的同时把令人担忧的方面抽象掉，比如低级别的线程，同步，线程安全和并发数据结构

总结性的语言，RxJava的本质就是“异步”。概括得很简洁和准确，但是同时也让很多初学者摸不着头脑，包括刚开始看的我自己。那么，这里应该要先介绍一下RxJava的基础，只有熟悉了，才能更好地了解。

RxJava基础概念

RxJava有四个基本概念：

  • Observable(可观察者，即被观察者)
  • Observer(观察者)
  • subscribe(订阅)
  • event(事件)

其中最核心的是Observable和Observer，Observable发出一系列事件，Observer通过subscribe()方法实现两者之间的订阅关系，从而Observer可以在需要的时候响应Obervable发出的事件。
RxJava跟设计模式中的观察者模式很类似，但是有一点明显不同，那就是如果一个Observerble没有任何的的Observer，那么这个Observable是不会发出任何事件的。

RxJava的好处

Android本身提供了Handler、AsyncTask用来处理异步操作，为什么我们还要

  • 简洁：程序的简洁对于异步操作来说很关键的，因为在调度过程比较复杂的情况下，异步代码经常会既难写也难被读懂。 Android自带的AsyncTask和Handler，其实都是为了让异步代码更加简洁，使得我们不用每次去new一个线程出来。RxJava 的优势也是简洁，但它的简洁的与众不同之处在于，随着程序逻辑变得越来越复杂，它依然能够保持简洁，什么复杂逻辑都能穿成一条线的简洁。
  • 线程容易切换：在开发过程当中，我们在做网络请求或者后台处理的时候，最重要的事情就是不能堵塞UI线程。RxJava是解决这类问题的好方法，因为它能够使得线程间的切换比较容易。

API介绍和原理简析

  • 扩展的观察者模式
    • 观察者模式
      A对象（观察者）对B对象（被观察者）的某种变化高度敏感，需要在B变化的一瞬间做出反应。在程序里面，我们采用注册(Register)或者称为订阅(Subscribe)的方式，告诉被观察者：我需要你的某某状态，你要在它变化的时候通知我。 Android开发中一个比较典型的例子是点击监听器OnClickListener 。对设置OnClickListener来说，View 是被观察者，OnClickListener是观察者，二者通过setOnClickListener()方法达成订阅关系。订阅之后用户点击按钮的瞬间，Android Framework就会将点击事件发送给已经注册的OnClickListener。采取这样被动的观察方式，既省去了反复检索状态的资源消耗，也能够得到最高的反馈速度。
    • RxJava的观察者模式
      与传统的观察者模式不同，RxJava的事件回调除了普通事件onNext()(相当于onClick()/onEvent)之外，还定义了两个特殊的事件：onCompleted和onError()
      • onCompleted():事件队列完结。
        RxJava不仅把每个事件单独处理，还会把它们看做一个队列。RxJava规定，当不会再有新的onNext()发出时，需要触发 onCompleted()方法作为标志。
      • onError(): 事件队列异常。
        在事件处理过程中出异常时，onError()会被触发，同时队列自动终止，不允许再有事件发出。

在一个正确运行的事件序列中, onCompleted() 和 onError() 有且只有一个，并且是事件序列中的最后一个。需要注意的是，onCompleted()和 onError()二者也是互斥的，即在队列中调用了其中一个，就不应该再调用另一个。

因为Subscriber实现了Observer，所以两者在使用方式上基本上是一样，而且，在subscribe过程中，Observer也总是会被转换成一个Subscriber再使用。如果是基本的使用，两者都是一样的，具体区别主要有两点：

  • onStart()
    这是Subscriber增加的方法。它会在subscribe操作刚开始，而事件还未发送之前被调用，可以做一些准备工作，例如数据的清零或重置。这是一个可选方法，默认情况下它的实现为空。需要注意的是，如果对准备工作的线程有要求（例如弹出一个显示进度的对话框，这必须在主线程执行），onStart()就不适用了，因为它总是在subscribe所发生的线程被调用，而不能指定线程。要在指定的线程来做准备工作，可以使用doOnSubscribe()方法。
  • unsubscribe()
    这是Subscriber所实现的另一个接口Subscription的方法，用于取消订阅。在这个方法被调用后，Subscriber将不再接收事件。一般在这个方法调用前，可以使用isUnsubscribed()先判断一下状态。unsubscribe()这个方法很重要，因为在 subscribe()之后， Observable会持有Subscriber的引用，这个引用如果不能及时被释放，将有内存泄露的风险。所以最好保持一个原则：要在不再使用的时候尽快在合适的地方(例如onPause()onStop()等方法中)调用unsubscribe()来解除引用关系，以避免内存泄露的发生。

    所有安装在手机上的应用都属于这一层次，包括自带的和应用市场下载的，当然也包括自己开发的程序

Android应用开发特色

  • 四大组件
    • 活动(Activity)：所有Android应用程序的门面，在应用中看到的大部分东西，都是直接或者间接放在活动中
    • 服务(Service)：无法直接看到，但是会在后台默默运行，即使退出应用，还是可以继续运行
    • 广播接收器(BroadcastReceiver)：允许应用接受各种广播消息，比如电话、短信等，自己开发的应用也可以向外发出广播
    • 内容提供器(Content Provider)：为了不同应用程序之间的数据共享提供可能，比如读取联系人
  • Sqlite数据库：Android系统自带的轻量级、运算速度快的嵌入式关系型数据库，支持标准SQL语法，也可以通过封装好的API进行数据操作，让数据交互变得更方便
  • 地理位置定位：相较于pc的一个亮点，手机内置的GPS，可以结合地图功能，做出许多有创意的应用
  • 强大的多媒体：音乐、视频、拍照、闹铃等多媒体服务，可以使应用丰富多彩
  • 传感器：重力传感器、加速度传感器、方向传感器等灵活使用，可以做出很多在pc上无法实现的应用

分析第一个Android程序

作为经典的第一个程序Hello World，这里不再针对创建流程做记录。但是呢，这里还是记录并分析一下第一个项目的目录结构

  • src：存放java代码
  • gen：系统自动生成，主要的是一个R.java文件，项目中添加的任何资源都会在其中生成一个对应的资源id，不能手动修改
  • assets：放一些随程序打包的文件，但是不会被压缩，例如静态网页、文本等
  • bin：编译时自动生成的文件，程序安装包就在里面
  • libs：放置一些第三方的jar包，此目录下的jar包会被自动添加到构建路径
  • res：存放图片、布局、字符串等资源，此目录下的所有文件都会在R.java中生成一个对应id
  • AndroidManifest.xml：项目配置文件，可以在里面声明权限、注册四大组件、指定程序兼容版本和目标版本等
  • project.properties：指定编译程序所使用SDK版本

为什么使用Log而不使用System.out

作为每个新手都很可能犯的错误，在这里有必要记录一下利弊

  • System.out.println()方便使用，日志打印不可控制、打印时间无法确定、不能添加过滤器、没有日志级别
  • Log使用简单，日志打印可控、有打印时间、可以添加过滤器、有日志级别

到此，《第一行代码》第一章的笔记内容就算是基本上整理完了，基本上内容都记录在博客中了，当然，还有一些比较重要，但是又没有在此记录的，比如说项目的运行机制、执行顺序、具体的资源文件分类等，或许以后会完善，或许会单独篇幅，或许会留在书中，哈哈哈！
